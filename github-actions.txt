Github Actions 
    workflow automation service by Github
    ex: CI/CD, Code & Repository Management    
CI/CD 
    Continuous Integration 
        automatically build & test code changes 
    Continuous Delivery 
        automatically deploy app
Workflow 
    attached to github repositories 
    contains one or more jobs 
    trigger upon events
    files in .github/workflows/
    Cancelling & Skipping Workflow Runs 
        Cancelling
            by default, workflows gets cancelled if job fails 
            by default, a job fails if at least one step fails 
            can cancel workflow manually
        Skipping
            if you add any of the following strings to the commit message in a push, 
            or the HEAD commit of a pull request
            [skip ci], [ci skip], [no ci], [skip actions], [actions skip]

Jobs 
    attached to workflow
    defines a Runner (execution environment)
    contains one or more steps 
    Run in parallel (default) or sequential
    can be conditional
    every job runs on it's own runner isolated from other jobs
    sequential jobs 
        needs: job-1 #or [job-1,job-2]
            job will run after job-1 completed successfully
Steps 
    attached to job 
    execute a shell script or an action
    can use custom or third-party actions
    steps are executed in order
    can be conditional

Workflow example
    name: My Workflow
    # define events using on, manual trigger event
    on: workflow_dispatch #or [workflow_dispatch, push]
    # define jobs 
    jobs:
        first-job: #job name 
            runs-on: ubuntu-latest
            steps:
                - name: print greeting 
                  run: echo "Hello"
                - name: print goodbye 
                  run: echo "Goodbye"
        
Events (Workflow Triggers)
    Repository
        push 
        pull_request
        create
            branch or tag created
        fork
        issues 
        issue_comment
            issue or pull_request comment action 
        watch 
            repo was starred 
    Other 
        workflow_dispatch   
            manually trigger workflow
        repository_dispatch 
            REST API request triggers workflow
        schedule 
            workflow is scheduled
        workflow_call
            can be called by other workflows 
    Activity Types 
        some events have activity types, more control over when to trigger 
        pull_request
            opened, closed, edited, assigned
            ex: pull_request:
                    types: 
                        - opened
            pull request from fork do not trigger a workflow (by default)
            first time contributers must be approved manually for workflow to run 
    Event Filters 
        more control over when to trigger
        push -> branch (target branch)
        ex: push:
                branches: 
                    - main 
                    - 'dev-*' # regex
    
Actions 
    custom application that performs a frequently repeated task 
    can build your own actions 
    official and community actions also present in Github marketplace 
    use uses keyword instead of run
    ex: steps:
            - name: Get Code 
              uses: actions/checkout@v3
            - name: Install NodeJS
              uses: actions/setup-node@v3
              with:
                node-version: '22'

Github Actions Context 
    collection of objects that provide metadata about a workflow run, 
    the repository, and the environment
    github
        The most common context, containing information about the workflow run 
        and the event that triggered it (e.g., commit SHA, branch name, event type).
    env
        Contains environment variables defined at the workflow, job, or step level.
    secrets
        Used to access sensitive repository secrets (like API tokens) without exposing them in logs.
    job
        Provides status and information about the currently running job.
    steps
        Contains outputs and statuses from steps that have already run within the same job.
    runner
        Includes details about the machine executing the job, such as its operating system and architecture.
    vars
        Stores custom configuration variables set at the organization, repository, or environment level.
    matrix
        Populated with the specific properties for a job when using a build strategy matrix.
    needs
        Used to access outputs from jobs that the current job depends on.

Job Artifacts 
    Output assets generated by the job 
    ex: app binary, website files etc 
    steps:
        - name: Upload build folder 
            uses: actions/upload-artifact@v3 
            with:
            name: dist-files 
            path: |
                dist
                package.json
    # another job 
    steps:
        - name: get build artifacts 
          uses: actions/download-artifact@v3 
          with: 
            name: dist-files
Job Outputs 
    Simple values that are needed by other jobs 
    outputs: 
        script-file: ${{ steps.publish.outputs.js-file }}
    steps:
       - name: Publish JS file name 
         id: publish
         run: find dist/assets/*.js -type f -execdir echo 'js-file={}' >> $GITHUB_OUTPUT ';'
    # another job 
    needs: build
    steps: 
        - name: Print JS file name 
          run: echo "${{ needs.build.outputs.script-file }}"
Dependency Caching 
    cache repeatitive step, to reuse by other jobs
    steps:
        # this step need to be present every job where we install deps
        - name: Cache dependencies 
          uses: actions/cache@v3
          with:
            path: ~/.npm
            key: deps-node-modules-${{ hashFiles('**/package-lock.json') }}
        
Environment Variables 
    can be given in workflow level, a job level and step level
    env:
        DB_NAME: db-demo
Secrets 
    can be added on repo level or organization level 
    github actions will redact secrets if you try to print
    env:
        DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
Github Repository Environments
    can provide environment specific secrets 
    required reviewers, special protection rules etc 
    jobs:
        deploy: 
            environment: staging
